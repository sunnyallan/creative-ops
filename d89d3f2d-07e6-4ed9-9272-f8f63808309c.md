# AI Creative Ops SaaS — Architecture & Build Plan
### Updated: June 2026 · Model stack: Gemini 3.1 Pro + Nano Banana

---

## What you're building

An AI-native creative operations SaaS that generates ads, marketing banners, emailers, and WhatsApp banners from a brief. The system learns continuously from human approvals, platform performance data, and persona feedback — producing increasingly optimised creatives over time.

## System architecture — 6 subsystems

### 1. Brand Setup (one-time per tenant)
**What it does:** User configures their brand and uploads assets before any creative work begins.

- User answers preset onboarding questions (brand name, target audience, brand values, tone of voice)
- Uploads logo files (SVG/PNG), font files (.ttf/.otf), and enters hex colour codes
- Signs a one-line asset permission declaration on upload (checkbox + timestamp, stored in DB)
- Defines one or more personas (age range, income tier, lifestyle descriptors, preferred imagery)
- All data stored in Supabase (Postgres with RLS keyed on `tenant_id`; assets in Supabase Storage under `/tenants/{tenant_id}/brand/`)

**Key design decision:** No rights management complexity. The platform stores whatever the user uploads; the user asserts they have rights. This is standard practice for design tools (Canva, Figma, etc.).

---

### 2. Intelligence Engine
**Orchestrator:** LangGraph (Python, MIT licence)
**Model:** `gemini-3.1-pro`
**Memory:** LangGraph `PostgresSaver` (short-term, per-campaign thread) + `PostgresStore` (long-term, cross-campaign learning)

**What it does:**
1. Reads the brand kit and persona definitions from Postgres
2. Runs persona × goal analysis: understands what content style, imagery, and copy tone suits each persona segment (e.g. fine-dining imagery + aspirational copy for high-income persona vs. value-led + pizza imagery for mass-market)
3. Produces a **structured JSON creative brief** — a list of banner requirements, each with: channel, dimensions, copy brief, persona segment, tone, image direction, call-to-action
4. Before finalising the brief, queries Qdrant (CLIP similarity search) to check whether a similar creative already exists — avoids regenerating identical work
5. **Learns over time:** rejection reasons from human review + platform performance metrics are written to `PostgresStore`, namespaced per tenant. Each new briefing cycle reads this store and adjusts

**Token optimisation:**
- Brand kit + system instructions placed at the start of every prompt (maximises Gemini implicit caching — 90% discount on cached prefix tokens for Gemini 3.x)
- Use Flash-Lite for persona classification steps; Pro only for brief generation and learning synthesis
- Batch overnight variant generation via the Gemini Batch API (50% discount)

---

### 3. Creative Engine
**Image generation:** Nano Banana API + fal.ai Flux fallback
**Copy generation:** `gemini-3.1-pro` / Flash-Lite
**Asset storage:** Supabase Storage + Qdrant (CLIP embeddings + metadata)

**What it does:**
1. Receives the JSON brief from Intelligence Engine
2. Generates copy (headlines, body, CTAs) per persona segment
3. Generates images via Nano Banana (primary) or fal.ai Flux Dev (bulk/LoRA-styled variants)
4. Composites creative: applies brand colours, fonts, and uploaded logos from the brand asset repo onto channel-specific templates:
   - WhatsApp banner: 1200×628px
   - Meta feed ad: 1080×1080px
   - Google display: 300×250, 728×90, 160×600
   - Emailer: 600px wide, HTML+inline CSS
5. Saves each output to Supabase Storage; generates CLIP embedding; writes metadata (brand_id, channel, persona, campaign_id, brief_id, creation_date, governance_status) to Qdrant
6. Future reuse: when a similar brief arrives, Qdrant similarity search returns past creatives — human reviewer can select existing instead of regenerating

**Brand asset repo approach:**
- Logos, fonts stored in Supabase Storage under tenant-namespaced paths
- No CDN or complex rights layer needed at MVP
- Creative Engine fetches assets at generation time via signed URLs

---

### 4. Governance Agent
**Stage 1 — Fast local NSFW:** `Falconsai/nsfw_image_detection` (ViT model, ~7MB, runs offline or via Replicate)
**Stage 2 — Cloud multi-category:** Sightengine API (`nudity-2.1`, `gore-2.0`, `weapon`, `offensive`, `text-content`)
**Stage 3 — Brand + hallucination judge:** `gemini-3.1-pro` vision

**What it does:**
1. Every generated creative passes through all three stages in sequence
2. Stage 1 is cheap and fast (offline); filters obvious NSFW before spending cloud API budget
3. Stage 2 catches embedded-text profanity (via OCR+NLP in Sightengine), weapons, hate symbols, gore
4. Stage 3: Gemini vision judge compares creative against the original brief and brand kit — checks brand colour adherence, logo presence/placement, copy accuracy, and hallucinated claims. Returns structured JSON: `{ pass: bool, issues: string[], severity: "block" | "warn" }`
5. **Fail → loop back:** Governance agent writes rejection reason back to LangGraph state; Creative Engine retries with the issue noted in the prompt context. Hard cap of 3 retries before escalating to human review queue with a flag.

---

### 5. Human-in-Loop — Review Dashboard
**Frontend:** Next.js (App Router, TypeScript, Tailwind, shadcn/ui)
**State persistence:** LangGraph `PostgresSaver` — human can approve hours later; the graph resumes from the last checkpoint

**What it does:**
1. Brand manager sees a grid of creatives pending review — thumbnail, channel label, persona tag, governance pass/warn status
2. **Approve:** Creative is queued for platform deployment
3. **Reject:** Manager types a reason (free text, optionally tagged: "wrong tone", "off-brand colour", "wrong imagery style", etc.). Rejection + reason written to LangGraph `PostgresStore` keyed to `(tenant_id, persona_id)` — feeds the Intelligence Engine's next briefing cycle
4. Batch actions available (approve all passed, reject all flagged)

---

### 6. Platform Deployment + Admin Dashboard
**Integrations:** Google Ads API, Meta Marketing API, WhatsApp Business Cloud API, emailer API (SendGrid / internal)
**Analytics:** Langfuse (open source LLM tracing + cost monitoring)
**Admin frontend:** Second role-gated section of the same Next.js app

**Deployment flow:**
1. Approved creatives are pushed to connected platforms via their respective APIs
2. Performance data (impressions, CTR, conversions) is ingested on a scheduled basis (Celery beat task) and written back to the `PostgresStore` keyed to the creative's brief and persona
3. Intelligence Engine reads this on its next cycle — closing the continuous learning loop

**Admin dashboard shows:**
- Per-tenant creative production volume and approval rates
- Platform performance by persona and channel
- Agent token usage and cost per campaign (Langfuse)
- API connection status (green/red per platform)
- Tenant billing status (Stripe)

## Full tech stack

### Frontend
| Layer | Choice | Why |
|---|---|---|
| Framework | Next.js 15 (App Router) | File-based routing, RSC, best-in-class DX |
| UI components | shadcn/ui + Tailwind CSS | Composable, accessible, no lock-in |
| Auth | Supabase Auth | Social login, magic link, JWT — one SDK |
| State / data fetching | TanStack Query + Zustand | Server state + client state cleanly separated |

### Backend
| Layer | Choice | Why |
|---|---|---|
| API | FastAPI (Python) | Async, typed, perfect for LangGraph integration |
| Agent orchestration | LangGraph | Purpose-built for stateful agentic loops, MIT licence |
| Task queue | Celery + Redis | Async generation jobs, scheduled data ingestion |
| DB | Supabase (Postgres + pgvector) | RLS for multi-tenancy, pgvector for lightweight fallback embeddings |
| Object storage | Supabase Storage | S3-compatible, same SDK, tenant-path isolation |
| Vector store | Qdrant | CLIP-native, rich payload filtering, Rust performance |
| Observability | Langfuse | LLM tracing, cost dashboards, open source |

### AI / Generation
| Role | Service | Package |
|---|---|---|
| LLM (all reasoning) | Gemini 3.1 Pro via Google AI | `google-genai` (Python) |
| Image gen primary | Nano Banana (Gemini 2.5 Flash Image) | `google-genai` |
| Image gen fallback | fal.ai Flux Dev | `fal-client` (Python) |
| CLIP embeddings | HuggingFace `clip-vit-base-patch32` | `transformers`, `torch` |
| NSFW local | Falconsai ViT | Replicate or local `transformers` |
| Brand safety cloud | Sightengine | `sightengine` (pip) |

## Phased build plan

### Phase 0 — Apply for APIs (start Day 1, runs in parallel)
- File Google Ads MCC + developer token application
- Create Meta Business app + start Business Verification
- Begin WhatsApp BSP onboarding
- These gate deployment, not development — start immediately

**Wall-clock time:** 2–6 weeks (external)

---

### Phase 1 — Core loop MVP (~6–8 weeks)
**Goal:** End-to-end: brief → image → governance → review queue

- [ ] Repo scaffold + docker-compose (Postgres, Qdrant, Redis, FastAPI, Next.js)
- [ ] Supabase project: schema, RLS policies, Storage buckets
- [ ] Brand setup UI: onboarding flow, asset upload with permission declaration checkbox
- [ ] LangGraph briefing agent: InMemorySaver → structured JSON brief output
- [ ] Gemini 3.1 Pro copy generation per persona
- [ ] Nano Banana image generation
- [ ] Channel template compositor (Pillow/Canvas for MVP; Fabric.js for web-based)
- [ ] Basic governance pipeline (Falconsai + Sightengine + Gemini vision judge)
- [ ] Next.js review dashboard: grid view, approve/reject with reason
- [ ] Qdrant: CLIP embedding + metadata write on every generated creative
- [ ] Swap InMemorySaver → PostgresSaver for persistent agent state

**Deliverable:** A single-tenant working prototype you can demo end-to-end.

---

### Phase 2 — Learning + Asset Recycling (~4–6 weeks)
**Goal:** The system actually gets smarter. Reuse reduces redundant generation.

- [ ] Rejection feedback loop: write to LangGraph PostgresStore, surface in next briefing cycle
- [ ] Qdrant similarity search in briefing step (deduplication before generation)
- [ ] Multi-persona brief generation (multiple segments in one campaign)
- [ ] Governance retry loop with hard cap + human escalation flag
- [ ] Langfuse integration: trace every LangGraph node, monitor token cost per campaign
- [ ] Admin dashboard v1: volume stats, approval rates, token costs

**Deliverable:** Demonstrable learning — second campaign for same brand noticeably better/faster.

---

### Phase 3 — Platform Integrations (~6–8 weeks, gated by Phase 0 approvals)
**Goal:** Approved creatives actually go live. Performance data flows back.

- [ ] Google Ads API integration (upload creative, create responsive display ad)
- [ ] Meta Marketing API integration (create ad creative, attach to campaign)
- [ ] WhatsApp Business Cloud API integration (send approved banner as template message)
- [ ] SendGrid emailer integration (render HTML email from approved emailer creative)
- [ ] Celery beat: scheduled performance data ingestion (CTR, impressions, conversions)
- [ ] Performance data → PostgresStore → Intelligence Engine learning cycle

**Deliverable:** Full loop: brief → creative → live on platforms → data back → better brief.

---

### Phase 4 — Multi-Tenant SaaS Hardening (~4–6 weeks)
**Goal:** Multiple paying customers, isolated, safe, billed.

- [ ] Postgres RLS: non-owner app role, `FORCE ROW LEVEL SECURITY`, all tables covered
- [ ] Qdrant: per-tenant collection or mandatory `brand_id` payload filter on all queries
- [ ] Supabase Storage: tenant-prefixed paths enforced in API layer
- [ ] Stripe subscription billing: plans (e.g. Starter / Growth / Scale) gating creative volume
- [ ] Rate limiting per tenant (API layer + Celery queue priority)
- [ ] Audit logs: every creative generation, governance result, human action
- [ ] Onboarding flow: self-serve signup → brand setup → first brief → first creative

**Deliverable:** Launchable SaaS product.

## Key decisions summary

| Decision | Choice | Rationale |
|---|---|---|
| Node-based pipeline | Dropped (n8n licensing trap for SaaS) | LangGraph in code is cleaner, fully licensable |
| Brand asset storage | User-uploaded repo + permission declaration | No rights complexity; standard for design tools |
| Image generation | Nano Banana primary, fal.ai Flux fallback | Gemini suite keeps the stack simple; Flux for cost/LoRA |
| Multi-tenancy | Postgres RLS + Supabase Storage paths | Security at the DB layer, survives any code bug |
| Memory | LangGraph PostgresSaver + PostgresStore | Persistent state enables long human review gaps |
| Asset recycling | Qdrant CLIP search | Text + image semantic search in one store |
| Observability | Langfuse | Purpose-built for LLM cost + trace monitoring |
| API approvals | File Day 1 in parallel | Google Ads + Meta are the true critical path |
